## Update — 2026-07-07 ~05:45 MDT: independent review ran; one blocker found + fixed

The operator ran the additional independent review. It surfaced **one `sev:block` install-path
defect** the publish-path checks structurally couldn't see — filed as **#25**, fixed on `feat/21`
the same session:

- **#25 — the registry archive installed dependency-less.** ComfyUI-Manager installs from
  `requirements.txt` and reads nothing from `pyproject`'s `[project] dependencies`; the repo had no
  `requirements.txt`, so every install `ModuleNotFoundError`'d on first load. **Fix (landed on
  `feat/21`):** a core-filtered `requirements.txt` (transformers/diffusers/accelerate; torch,
  torchvision, numpy, Pillow omitted as core-provided wheel/ABI hazards); `tests/test_requirements_sync.py`
  as the drift-guard (pyproject → requirements.txt is a *test*, not a convention); and a front-door
  `transformers==5.13.0` version guard in `dgemma/model.py` (Manager silently skips downgrade pins,
  so a wrong version can slip through → actionable error, not a raw traceback). Fast suite **136
  passed**, no regressions. Open guard-strictness question: the version check is exact `==5.13.0`;
  loosening to patch-tolerant `5.13.x` is an open one-line call.

Docs also grew this session: `VISION.md` (verified arXiv citations), a loud hardware/memory
requirements section, and README/CONTRIBUTING polish.

**Publish gate now:** #25 was the review's only blocker and it's resolved on `feat/21`. The gate is
otherwise clear; publish remains held on the operator's word (merge PR #24 = publish 0.2.0).
